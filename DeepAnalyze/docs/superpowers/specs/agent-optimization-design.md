# DeepAnalyze Agent 优化设计原则（v2）

> 基于实际工具测试 + 系统能力演进，更新于 2026-05-09

---

## 核心设计哲学

1. **通用性优先**：Agent 核心系统对任何领域、任何类型的数据保持通用。领域优化通过 Skill/Plugin 实现。
2. **弹性执行**：所有约束都是弹性的——模型根据任务复杂度自动调整，不做固定截断/超时/轮次限制。
3. **分层构建**：能力按层级组织，低层解决不了的问题才升级到高层。
4. **零幻觉**：所有事实性陈述必须基于工具返回的数据，不得使用模型自身记忆补充。
5. **数据驱动**：工具引导基于每个工具的真实能力和局限性，而非理想化描述。

## 分层架构

### L1：工具层（Tool Layer）

每个工具的实际能力、局限性和适用场景。

#### 检索类工具真实能力评估

| 工具 | 数据源 | 覆盖率 | 精确度 | 适用场景 | 已知局限 |
|------|--------|--------|--------|---------|---------|
| `wiki_browse(listDocuments=true)` | documents 表 | **100%** | 高 | 获取完整文档清单和 L0 摘要，**任何分析的第一步** | 摘要截断至 300 字 |
| `run_sql` | PostgreSQL | **100%** | 最高 | 统计、聚合、精确筛选、数据探查 | Agent 需知表结构 |
| `doc_grep` | wiki_pages 表 | **100%**（受 maxResults 截断影响） | 精确（正则） | 关键词/精确匹配搜索 | `totalMatches` 被截断；默认 maxResults=30 偏低 |
| `expand` | wiki_pages 表 | 单文档 100% | 高 | 获取单个文档完整 L1/L2 内容 | 批量模式在数据不一致时静默失败 |
| `kb_search` | FTS + 向量检索 | **~4%**（top-10/239） | 中等（语义近似） | 快速发现主题相关文档，**绝不能依赖它保证完整性** | 分数不可跨查询比较；topK 硬上限 |
| `glob` | 磁盘文件 | **100%**（计数）/ 200文件上限（列表） | 高 | 统计文件数量、发现文件类型 | `files` 数组截断至 200 条 |
| `grep` | 磁盘文件（.md/.txt/.csv/.json/.yaml/.yml） | 覆盖 6 种文本格式 | 高（正则） | 搜索磁盘原始文件 | 不覆盖 .py/.html/.xml 等；10s 超时 |
| `read_file` | 磁盘文件 | 100% | 最高 | 直接读取任意文本文件（含预处理产物） | 仅磁盘，不读数据库 |

#### 工具能力分层原则

**可靠性分层**（按数据完整性排列，非使用优先级）：
1. `run_sql` / `wiki_browse` — 100% 覆盖，精确完整
2. `doc_grep` + `grep` — 精确文本匹配，互补覆盖（数据库 vs 磁盘）
3. `expand` / `read_file` — 单文档完整内容（数据库 vs 磁盘，数据一致但来源不同）
4. `bash` — 完整文件系统访问，批量操作效率高，可编程处理
5. `kb_search` — 语义发现，topK 限制导致必然遗漏

**注意**：这不是使用优先级。Agent 应根据任务需要自主选择最合适的工具。bash/grep/read_file 与 expand/doc_grep/run_sql 是互补关系，不是从属关系。

#### L1 工具问题待修复

- [ ] `doc_grep` 的 `totalMatches` 应返回真实总数而非截断数
- [ ] `glob` 应在返回中增加 `truncated: true` 标记
- [ ] `expand` 批量模式失败时应提供更准确的诊断信息
- [ ] `kb_search` 描述应明确说明 topK 限制和分数不可比较性

### L2：引导层（Harness/Prompt Layer）

#### 2.1 工具引导原则

**核心原则：描述能力边界，不规定使用顺序**

工具引导不应规定"先用 A 再用 B"的固定流程，而是：
- 描述每个工具能做什么和不能做什么
- 让 Agent 根据任务自主判断最合适的工具组合
- 在关键场景给出示例而非硬性流程

**严格禁止的写法**（在 `tool-guidance.ts` 和 `tool-setup.ts` description 中）：
- ❌ "优先使用 X" — 偏袒特定工具
- ❌ "适合其他工具无法覆盖的场景" — 将工具降级为兜底
- ❌ "请谨慎使用" — 主观推荐
- ❌ "A → B → C" 固定搜索顺序 — 规定流程
- ❌ "不要用 bash 的 python/json 解析文件" — 限制合法操作
- ❌ 在 execute() 中硬拦截合法命令（如 json.loads）— 代码级限制

**正确的写法**：
- ✅ "grep 可批量搜索全部文件；python3 可编程处理任意数据" — 能力描述
- ✅ "只搜索 wiki_pages 已编译内容，不含磁盘原始文件" — 局限性描述
- ✅ "topK=10 必然遗漏大量相关文档" — 诚实评估
- ✅ 每个工具同时列出强项和弱项（工具诚实评估表）

**bash 的特殊地位**：bash 是一等公民工具，拥有完整文件系统访问能力。它不是"兜底工具"或"最后手段"，而是与 expand/doc_grep/run_sql 平等的工具选择。Agent 应能自由选择用 `grep -rl '关键词' wiki/` 还是 `doc_grep` 来搜索——两种方式互补，各有适用场景。

**工具能力表结构**：每个工具必须包含：
1. 数据源和覆盖率（客观事实）
2. 特点描述（能力描述，不包含推荐）
3. 在"工具诚实评估"表中同时列出强项和弱项

#### 2.2 反幻觉层级

三层反幻觉（保持现有设计，已经很好）：
- `basic`：基础引用要求（通用/探索 Agent）
- `standard`：结构化验证要求（编译/报告 Agent）
- `strict`：最高严谨度（验证 Agent、预处理 Skill）

#### 2.3 提示词构建

保持 `SystemPromptBuilder` 的静态/动态分离设计。动态部分包括：
- 可用技能列表（含高成本标记）
- 知识库范围
- 会话记忆

### L4：Agent 循环层（Agent Loop）

#### 4.1 弹性轮次管理

保持 `calculateDynamicTurnBudget()` 设计：
- 简单=15, 中等=40, 复杂=100（建议值）
- 硬限制 = 3x 建议值（最低 100）
- 80% 建议值时注入收尾引导

**原则**：不在 Skill 中设硬性 `maxTurns`。Skill 是经验指导，不是执行限制。

#### 4.2 输出 token 恢复

4 级升级（已实现）：base → 2x → 4x → 65K → 131K，最多 3 次恢复。

#### 4.3 卡住检测

`StuckDetector` 检测 4 种模式，已验证有效。保持弹性——允许 `expand`/`kb_search` 批量操作豁免。

### L5：多 Agent 编排层（Multi-Agent Orchestration）

保持现有设计（5 种调度模式 + DAG + Council）。

新增要点：
- 子 Agent 动态超时（基于任务复杂度）保持弹性

### L6：场景层（Skill/Plugin）

#### 6.1 Skill 分类

| 类型 | 特征 | 示例 | 触发方式 |
|------|------|------|---------|
| 通用增强 | 提升特定场景的方法论 | `deep-research`, `chunked-analysis` | Agent 自主判断 |
| 按需触发 | 高成本，仅用户明确要求 | `知识库预处理` | `[高成本/按需触发]` 标记 + 确认门控 |
| 通用工具化 | 日常分析可复用 | `文档摘要`, `对比分析`, `表格专项分析` | Agent 自主判断 |

#### 6.2 Skill 防护机制

- 高成本 Skill 用 `[高成本/按需触发]` 标记
- `skill_invoke` 工具拦截高成本 Skill，返回确认请求
- Agent 系统提示词中用 🔴 标记高成本 Skill

#### 6.3 Plugin 原则

- 领域特定内容通过 Plugin 实现，不进入核心系统
- Plugin = 一组 Skills + 自定义 Agent + 领域配置
- 判断标准：修改只对一个领域有效 → Plugin；对所有领域有效 → 通用 Skill 或核心系统

## 新增：文件系统优先原则

### 背景

知识库数据现在以完整的目录文件结构存储在磁盘上：

```
data/
├── wiki/{kbId}/
│   ├── manifest.json          # KB 清单
│   ├── documents/{docId}/
│   │   ├── parsed.md          # L1 解析内容
│   │   ├── raw/
│   │   │   ├── docling.json   # L2 原始数据
│   │   │   └── metadata.json  # 处理元数据
│   │   ├── abstract.md        # L0 摘要
│   │   ├── overview.md        # L1 概览
│   │   ├── structure/
│   │   │   ├── image.md       # L1 结构化内容
│   │   │   └── ...
│   │   └── ...
│   └── _preprocessing/
│       ├── overview.md        # 全局概览
│       ├── tables/*.csv       # 还原的表格
│       ├── image_audit.md     # 图片质量审计
│       └── annotation_audit.md
├── original/{kbId}/           # 原始上传文件
```

Agent 可以通过 `read_file`、`bash`、`glob`、`grep` 直接访问这些文件，就像 Claude Code 操作文件系统一样。

### 原则

1. **文件系统是第一类数据源**：`read_file`、`glob`、`grep`、`bash` 可以直接高效地访问知识库内容，不经过数据库中间层。
2. **数据库和文件系统互补**：`run_sql`、`wiki_browse`、`doc_grep`、`expand` 查询数据库（有索引、可聚合）；`read_file`、`grep` 查询文件系统（更直接、更灵活）。两者数据一致但不等价。
3. **不偏袒任何路径**：Agent 根据任务自主选择。统计聚合 → `run_sql`；精确搜索 → `doc_grep`/`grep`；直接阅读 → `read_file`/`expand`；发现相关文档 → `kb_search`（注意覆盖率限制）。
4. **预处理数据自动可见**：`wiki_browse(listDocuments=true)` 自动附带预处理产物信息，Agent 无需额外步骤发现。

## 不变的核心约束

以下原则在任何优化中都不改变：

1. **零幻觉**：事实性内容必须有工具数据支撑
2. **通用性**：核心系统不包含领域特定内容
3. **弹性**：不做固定截断、固定超时、固定设计
4. **分层**：优化放入正确的层级
5. **Agent 自主性**：系统提供信号，不设约束
