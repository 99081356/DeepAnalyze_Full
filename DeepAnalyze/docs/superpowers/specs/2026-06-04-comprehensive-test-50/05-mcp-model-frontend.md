## T37: MCP服务器增删与工具发现

### 测试设计
**操作**（通过前端设置面板 + API）：
> 1. 在MCP配置页面添加一个新的stdio类型MCP服务器
> 2. 点击连接，验证工具发现
> 3. 在对话中尝试使用MCP提供的工具
> 4. 断开MCP服务器
> 5. 删除MCP服务器
> 6. 验证工具已移除

### 观察目标
1. **添加成功**：MCP服务器出现在列表中，状态为"未连接"
2. **连接成功**：点击连接后状态变为"已连接"，显示发现的工具数量
3. **工具注册**：MCP工具以 `mcp__serverName__toolName` 格式注册到ToolRegistry
4. **工具可用**：Agent可以调用MCP工具并得到正确结果
5. **断开安全**：断开后工具不可调用，不崩溃
6. **删除清理**：删除后ToolRegistry中不再有该MCP的工具
7. **前端面板**：MCPServersPanel正确显示所有状态变化

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 连接失败 | 检查MCP client的transport实现（stdio/sse/websocket） |
| 工具不注册 | 检查tool discovery的回调注册到ToolRegistry |
| 断开后崩溃 | 添加MCP工具调用前的连接状态检查 |
| 删除不清理 | 检查DELETE /mcp/:id的清理逻辑 |

## T38: 主模型实时切换

### 测试设计
**操作**（通过前端设置面板）：
> 1. 查看当前主模型配置（默认minimax-text）
> 2. 切换主模型为其他可用模型（如qwen或glm）
> 3. 在对话中发送一个问题，验证使用新模型
> 4. 切换回原模型
> 5. 再次发送问题，验证恢复

### 观察目标
1. **配置页面**：ModelsPanel正确显示当前配置和可选模型
2. **切换生效**：切换后新session使用新模型（通过日志验证modelId）
3. **热更新**：不需要重启服务即可生效（ModelRouter的bumpConfigVersion）
4. **旧session不受影响**：已运行的session继续用旧模型
5. **回切正常**：切换回原模型后功能完全正常
6. **测试连接**：每个模型的"测试连接"按钮返回正确状态
7. **前端显示**：配置页面正确反映当前模型名称和参数

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 切换不生效 | 检查ModelRouter的config version检测和热更新 |
| 旧session受影响 | 确认model选择在run级别，不在全局级别 |
| 测试连接失败 | 检查provider test API和connectivity检测 |
| 前端不同步 | 检查ModelsPanel的数据刷新和API调用 |

## T40: Provider熔断与降级

### 测试设计
**操作**：
> 1. 配置主模型和一个辅助模型
> 2. 模拟主模型连续失败（如设置错误的API key）
> 3. 观察系统是否自动降级到辅助模型
> 4. 修复主模型配置
> 5. 观察是否恢复到主模型

### 观察目标
1. **熔断触发**：连续3次失败后Circuit Breaker打开
2. **自动降级**：熔断后自动切换到辅助模型
3. **降级提示**：前端显示降级信息（toast或状态提示）
4. **半开恢复**：60s后自动尝试恢复主模型（半开状态）
5. **恢复后正常**：主模型恢复后正常使用
6. **无服务中断**：降级期间Agent仍可响应（使用辅助模型）

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 不熔断 | 检查Circuit Breaker的阈值和计数逻辑 |
| 不降级 | 检查fallback provider选择逻辑 |
| 全部失败 | 确保有合理的错误提示，不是静默失败 |
| 不恢复 | 检查half-open状态的探测机制 |

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 不流式 | 检查SSE的content_delta事件发送和前端处理 |
| 格式错乱 | 检查Markdown渲染和streaming chunk边界处理 |
| 不滚动 | 检查auto-scroll逻辑（useEffect监听content变化） |
| 刷新丢失 | 检查SSE恢复机制和draft message持久化 |
| thinking不停 | 检查finish事件的处理和thinking状态重置 |

## T44: 多面板切换与状态保持

### 测试设计
**操作**（前端交互）：
> 1. 打开聊天页面，发送一条消息
> 2. 切换到知识库页面，浏览文档
> 3. 切换到设置页面，修改模型参数
> 4. 切换到报告页面，查看报告
> 5. 切回聊天页面
> 6. 验证聊天消息和Agent状态是否保持

### 观察目标
1. **路由切换**：所有页面路由正常切换，无404
2. **状态保持**：切回聊天页面时，之前的消息仍显示
3. **Agent状态保持**：如果Agent还在运行，切回后继续显示进度
4. **设置持久化**：修改的设置在切换后保持
5. **WebSocket保持**：页面切换不导致WebSocket断开
6. **无内存泄漏**：反复切换不导致浏览器内存持续增长
7. **响应速度**：页面切换响应在500ms内

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 状态丢失 | 检查Zustand store的持久化和组件卸载时的清理 |
| Agent中断 | 检查路由切换时的SSE/WebSocket管理 |
| 设置丢失 | 检查settings API的持久化 |
| WS断开 | 检查useWebSocket在不同路由下的连接管理 |
