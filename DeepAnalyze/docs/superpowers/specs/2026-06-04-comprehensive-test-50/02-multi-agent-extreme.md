## T11: 并行模式——10子Agent大规模文档分组分析

### 测试设计
**知识库**：bigtest（242个文档）
**提示词**：
> 请使用workflow_run并行模式，将知识库中所有文档按类型分成5-8组，每组分配一个子Agent进行深度分析。每个子Agent负责：文档清单整理、核心内容摘要、关键发现提炼。所有子Agent完成后，由主Agent汇总输出综合分析报告。

### 观察目标
1. **workflow_run触发**：Agent主动调用workflow_run(mode=parallel)，而非逐个delegate_task
2. **分组合理性**：文档按逻辑关联分组（如论文一组、剧本杀按剧本分组），不是随机分配
3. **子Agent数量**：5-8个子Agent并行运行，面板正确显示所有子Agent进度
4. **SubAgentPanel显示**：
   - 折叠态：显示所有子Agent名称和状态chips
   - 展开态：每个子Agent显示执行时间、工具调用数、状态
   - 完成态：全部显示"已完成"，无卡死
5. **结果完整性**：每组都有分析结果，无遗漏组
6. **综合报告质量**：主Agent汇总报告综合了各组的核心发现，不是简单拼接
7. **无重复**：不同组的分析内容不重叠

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 未触发workflow_run | 改进agent-definitions.ts中的委托引导 |
| 子Agent卡死 | 检查sub-agent timeout和stuck detection |
| 面板不显示 | 检查workflow事件SSE推送和前端处理 |
| 结果遗漏 | 检查persistAgentOutput和结果收集逻辑 |

## T12: Pipeline模式——串行链式分析流水线

### 测试设计
**知识库**：lbctest
**提示词**：
> 请使用workflow_run的pipeline模式，按以下步骤串行分析案件：
> 1. 信息收集Agent：搜集所有文档，按类别（诉讼文书/证据/笔录/物证）分类
> 2. 事实提取Agent：从分类结果中提取所有事实要素（人物/时间/地点/金额/行为）
> 3. 证据链构建Agent：基于事实要素构建完整证据链
> 4. 法律分析Agent：基于证据链进行法律分析和量刑建议
> 5. 报告生成Agent：生成最终结构化报告
> 每个Agent的输出作为下一个Agent的输入。

### 观察目标
1. **串行执行**：5个Agent严格按顺序执行，前一个完成后才开始下一个
2. **上下文传递**：后续Agent能看到前序Agent的输出结果（通过context传递）
3. **Pipeline面板**：前端正确显示流水线进度（Agent 1完成→Agent 2开始→...）
4. **信息无损传递**：关键事实从第2步传递到第5步无丢失
5. **每步输出质量**：每步都有实质性输出，不是"见上文"
6. **最终报告**：报告涵盖所有5步的核心内容

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 非严格串行 | 检查pipeline模式实现，确保wait for predecessor |
| 上下文丢失 | 增大context传递的token预算，改用文件而非context传递 |
| 中间步骤空输出 | 添加每步必须输出检查，空输出触发重试 |

## T13: Council模式——多角度+交叉审查

### 测试设计
**知识库**：bigtest（聚焦论文部分）
**提示词**：
> 请使用council模式，安排3个不同视角的子Agent分析知识库中的学术论文：
> - Agent A（技术评估者）：评估每篇论文的技术创新性和方法有效性
> - Agent B（应用分析者）：分析每篇论文的实用价值和落地场景
> - Agent C（批判审查者）：找出每篇论文的局限性和潜在问题
> 第一轮完成后进行交叉审查，每个Agent审查另一个Agent的分析。

### 观察目标
1. **两轮执行**：第一轮3个Agent并行分析，第二轮交叉审查
2. **视角差异化**：三个Agent的分析角度确实不同（不是重复相同内容）
3. **交叉审查质量**：审查Agent指出被审查分析的不足或遗漏
4. **Council面板**：显示Phase 1和Phase 2两个阶段
5. **审查有实质**：交叉审查不是走过场，有具体的修正和补充
6. **最终综合**：主Agent综合三轮分析得出最终结论

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 只执行一轮 | 检查council模式的Round 2触发逻辑 |
| 视角雷同 | 改进per-agent system prompt差异化 |
| 审查空洞 | 添加审查质量检查：要求指出至少N个不足 |

## T14: Graph模式——DAG依赖关系分析

### 测试设计
**知识库**：bigtest
**提示词**：
> 使用graph模式分析知识库。设置以下Agent依赖关系：
> - 基础研究Agent（无依赖）：浏览所有文档，提取基本信息
> - 论文分析Agent（依赖基础研究）：深度分析论文部分
> - 剧本杀分析Agent（依赖基础研究）：深度分析剧本杀部分
> - 多媒体分析Agent（依赖基础研究）：分析图片/音频/视频
> - 综合报告Agent（依赖论文分析+剧本杀分析+多媒体分析）：汇总所有分析结果

### 观察目标
1. **DAG拓扑正确**：基础研究先执行，三个分析Agent并行执行，综合报告最后执行
2. **依赖等待**：综合报告Agent确实等所有三个分析Agent完成后才开始
3. **面板显示**：SubAgentPanel正确显示依赖关系和执行顺序
4. **结果引用**：综合报告引用了三个分析Agent的具体发现
5. **无死锁**：所有Agent都能正常完成，不存在循环依赖

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 拓扑错误 | 检查graph模式的dependsOn解析和拓扑排序 |
| 综合报告未等 | 检查依赖满足检测逻辑 |
| 面板顺序错乱 | 改进前端对DAG执行顺序的显示逻辑 |

## T15: 超长工具调用链（50+步）

### 测试设计
**知识库**：bigtest
**提示词**：
> 请逐个阅读知识库中所有PDF论文的L1内容，对每篇论文输出以下信息：
> 1. 论文标题和作者
> 2. 核心方法（200字以内）
> 3. 实验结果中的关键指标（精确引用数值）
> 4. 主要局限性
> 每篇论文的分析必须通过expand工具获取L1内容后再总结。

### 观察目标
1. **工具调用次数**：至少50次工具调用（16篇论文 × expand + kb_search + 汇总）
2. **无中断**：整个执行过程不中断，不触发maxTurns限制
3. **stuck detection不误触发**：批量expand操作不被误判为stuck
4. **每步质量**：每篇论文的分析质量一致，不会因为轮次增加而降低
5. **context管理**：观察context compaction何时触发，压缩后是否保留对已处理论文的记忆
6. **最终汇总完整**：所有论文都在汇总中出现，无遗漏

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 中途中断 | 增大dynamic turn budget，确认estimateTaskComplexity给出complex/hard |
| stuck误判 | 确保expand批量操作在stuck detection的exempt列表中 |
| 压缩丢记忆 | 改进post-compact re-injection，注入已完成部分的摘要 |
| 后期质量下降 | 这是模型注意力问题，可通过分段汇总+最终综合来缓解 |

## T16: 主Agent+子Agent混合工具调用

### 测试设计
**知识库**：lbctest
**提示词**：
> 分析这个案件。先由你（主Agent）搜索和阅读关键文档，形成初步理解。然后派出2个子Agent分别分析不同的证据组。子Agent完成后，由你综合所有信息输出最终分析。

### 观察目标
1. **主Agent工具调用**：主Agent先使用kb_search/expand等工具建立理解
2. **子Agent分派**：主Agent正确使用delegate_task分派子任务
3. **子Agent工具调用**：子Agent也使用kb_search/expand等工具
4. **工具权限隔离**：子Agent不使用workflow_run/push_content等被阻止的工具
5. **结果回收**：主Agent收到子Agent的返回结果并整合
6. **最终输出**：综合报告包含主Agent和子Agent的所有发现

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 主Agent不分派 | 改进委托引导，在适当场景推荐delegate_task |
| 子Agent权限错误 | 检查sub-agent blocked tools配置 |
| 结果丢失 | 检查delegate_task的结果收集和注入逻辑 |

## T17: Agent失败+部分成功混合场景

### 测试设计
**知识库**：bigtest
**提示词**：
> 请使用workflow_run并行模式，分5个子Agent分析知识库中的不同部分：
> - Agent 1：分析所有PDF论文
> - Agent 2：分析"5241 自杀派对"剧本杀
> - Agent 3：分析所有图片
> - Agent 4：分析Excel表格
> - Agent 5：分析所有音频和视频
> 如果某个子Agent失败，其他Agent应继续执行并输出结果。

### 观察目标
1. **容错性**：部分Agent失败不影响整体工作流
2. **失败处理**：失败的Agent在面板显示"失败"状态（红色），不是卡在"运行中"
3. **成功Agent完整**：成功的Agent结果完整保存和推送
4. **综合报告**：主Agent汇总时注明哪些部分因Agent失败而缺失
5. **前端展示**：面板同时显示成功（绿色）和失败（红色）的Agent

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 一个失败全部中断 | 检查workflow-engine的per-agent error isolation |
| 失败Agent卡住 | 确保failure事件正确发送到前端 |
| 结果丢失 | 检查persistAgentOutput在部分失败时的行为 |

## T18: 嵌套工作流——子Agent中再分派子Agent

### 测试设计
**知识库**：bigtest
**提示词**：
> 请将知识库按主题分为3个大组，每组使用一个子Agent分析。其中分析论文组的子Agent如果发现论文太多，应该再使用delegate_task分派更细粒度的分析任务。

### 观察目标
1. **嵌套触发**：子Agent主动使用delegate_task进一步分派
2. **嵌套深度**：至少2层嵌套（主→子→孙）
3. **层级限制**：不会无限递归，有合理的深度控制
4. **结果汇总链**：孙Agent结果→子Agent→主Agent，信息传递完整
5. **面板显示**：前端能正确显示多层嵌套关系
6. **性能**：嵌套不会导致资源耗尽（连接池/内存/并发数）

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 嵌套不支持 | 检查sub-agent中delegate_task是否被正确阻止或放行 |
| 无限递归 | 添加嵌套深度限制（如max depth=3） |
| 面板混乱 | 改进SubAgentPanel对嵌套层级的可视化 |
| 资源耗尽 | 检查max concurrent agent runs限制（默认8） |

## T19: 超长工作流——100+步综合任务

### 测试设计
**知识库**：bigtest + lbctest（双库绑定）
**提示词**：
> 请完成以下综合任务：
> 1. 浏览bigtest知识库所有文档，按类型分类
> 2. 对每篇论文展开L1内容并分析技术要点（16篇，每篇至少3次工具调用）
> 3. 对每个剧本杀进行完整剧情分析和推理（4个剧本杀，每个至少5次工具调用）
> 4. 浏览lbctest知识库所有文档，分类整理
> 5. 提取lbctest中所有时间线信息
> 6. 构建lbctest的完整证据链
> 7. 交叉对比两个库中法律相关内容的异同
> 8. 生成最终综合报告

### 观察目标
1. **总步骤数**：100+次工具调用（expand×20 + search×10 + read×20 + bash×10 + 其他×40）
2. **跨库切换**：Agent正确处理两个知识库的切换，不混淆数据
3. **长时间运行**：整个任务持续运行不中断（可能需要30分钟以上）
4. **context管理**：经历多次context compaction后仍保持关键信息
5. **最终报告完整**：覆盖所有8个子任务的核心内容
6. **性能不衰减**：后期工具调用的质量和准确度不下降

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 中途停止 | 检查maxTurns和timeout设置，确认dynamic budget给出足够轮次 |
| 库混淆 | 改进多KB场景的session scope管理 |
| 后期质量差 | 改进context compaction的关键信息保留策略 |
| 超时 | 调整stream idle timeout和agent总超时 |

## T20: 取消工作流并恢复

### 测试设计
**知识库**：bigtest
**提示词**：先发送一个需要长时间运行的分析请求，在运行过程中（等待3-5个工具调用后）手动取消任务。取消后立即发送一个新的简单问题。

### 观察目标
1. **取消生效**：取消后Agent立即停止，不再执行新的工具调用
2. **资源清理**：子Agent（如果有）也被取消，不继续消耗资源
3. **状态一致**：前端正确显示"已取消"状态
4. **新任务正常**：取消后发送新问题，Agent正常响应，不受前一次取消影响
5. **会话状态**：会话消息列表完整显示取消前的对话+取消消息+新对话
6. **无内存泄漏**：取消后系统资源正常释放

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 取消不生效 | 检查AbortController传递和各层cancel传播 |
| 新任务异常 | 检查cancel后的状态重置（session清理） |
| 资源未释放 | 检查workflow engine的cleanup逻辑 |
