## T01: bigtest全库综合深度分析（超长输出）

### 测试设计
**知识库**：bigtest（242个文档）
**提示词**：
> 现在请你分析知识库文档的全部内容，分析清楚所有内容的主要分类关系、从属关系，分析清楚每一类、每一块知识的相关性。针对论文，请给出一份详细的分析报告分析这些论文的技术演进关系，分析不同技术的优劣势，预测未来核心的研究方向和建议，给出详细的分析报告。如果是剧本杀，请分析整个剧本的关系，找出明确的杀手和时间线推理关系，找出所有证据链条和推理逻辑链条，每个剧本杀单独给出详细的完整故事脉络和逻辑关系推理。如果是表格，详细统计分析表格内容和数据情况。其他类型也自定义不同需求，对整个知识库进行全面深入完整的分析与整理。

### 观察目标
1. **分类完整性**：所有242个文档都被覆盖，无遗漏类别（论文/剧本杀/图片/音视频/表格/代码等）
2. **独立报告推送**：每类分析作为独立push_content卡片推送，卡片之间无内容重复
3. **报告体量**：总输出不少于5万字，每类至少3000字详细分析
4. **无幻觉**：
   - 论文分析中引用的技术名称、作者、年份必须能在文档中找到对应
   - 剧本杀推理中的"凶手"判定必须有明确证据支持，不能凭空推断
   - 表格统计数据（行数、列数、样本值）与原文一致
5. **结构清晰**：每类有独立标题、子标题、表格/列表组织
6. **前端显示**：所有push_content卡片正确渲染，可滚动查看完整内容，无截断

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 遗漏类别 | 改进wiki_browse引导，要求先用listDocuments获取完整清单再分组 |
| 内容重复 | 改进push_content去重机制，添加内容指纹检测 |
| 输出太短 | 调整max_output_tokens恢复策略，确保长输出续写不中断 |
| 幻觉问题 | 加强anti-hallucination提示，要求所有事实引用doc_id |
| 前端截断 | 检查push_content卡片虚拟滚动，增大渲染缓冲区 |

## T02: lbctest全案司法分析（证据链+时间线+量刑）

### 测试设计
**知识库**：lbctest（110个文档）
**提示词**：
> 1. 详细分析整个案件，按人、时、地、事、物的关键信息梳理，整理完整的案件介绍。
> 2. 还原整个案件完整的时间线关系，给出所有案件推理和证据证明材料。
> 3. 梳理资金关系和流向，明确分析资金转移情况，核对受害人的资金损失情况。
> 4. 给出量刑标准和法规指引，并对每个量刑项目给出全部证据关联材料信息。
> 5. 使用司法证据链标准的skill，输出符合司法证据链要求的文档报告。

### 观察目标
1. **案件人物完整**：所有涉案人员（嫌疑人、受害人、证人等）均被列出，无遗漏
2. **时间线连贯**：时间线覆盖案件全过程，每个时间节点有对应文档证据支撑
3. **资金流向清晰**：资金转移路径明确，金额与原文一致（零容忍幻觉——数字错误即失败）
4. **量刑标准准确**：引用的法条条文号和量刑区间须与中国现行刑法条文一致
5. **证据链完整**：使用evidence-chain skill后，每条证据有`da-evidence://`链接，可点击跳转查看原文
6. **无幻觉**：所有金额、日期、人名、地址与原文严格一致，不得编造
7. **前端展示**：证据链接可点击、跳转到原文档对应位置，蓝色高亮标注

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 证据链断裂 | 优化evidence-chain skill的搜索维度，增加金额/日期维度的kb_search |
| 法条错误 | 验证模型是否具备最新刑法知识，考虑添加法规知识库 |
| 证据链接不可点击 | 检查da-evidence://协议前端解析和路由 |
| 人物遗漏 | 改进entity-network skill，要求穷举搜索所有人名 |

## T03: 超长单文档深度分析（20万字级输出）

### 测试设计
**知识库**：bigtest
**提示词**：
> 请选择知识库中的"5241 自杀派对"剧本杀（79个文档），对每个角色的剧本、线索、结局进行逐字逐句的深度分析。要求：
> 1. 列出所有角色的完整信息（姓名、身份、动机、行为时间线）
> 2. 逐条分析每条线索的含义、指向、与其他线索的关联
> 3. 构建完整的推理链：从初始线索到最终结论的每一步推理过程
> 4. 对每个可能的假设进行验证或排除
> 5. 输出完整的故事还原，包括每个角色的视角
> 目标输出20万字以上的完整分析报告。

### 观察目标
1. **输出体量**：最终输出不少于15万字（允许20%容差）
2. **长输出续写**：观察finish_reason=length的续写机制是否正常触发，续写内容是否衔接自然
3. **上下文管理**：观察context compaction是否在关键时刻触发，压缩后是否丢失关键信息
4. **内容连贯性**：续写前后的内容逻辑连贯，无断层、无重复段落
5. **角色覆盖**：所有角色都有独立的深度分析，无遗漏
6. **线索覆盖**：所有线索卡片都被分析，每条线索至少被引用一次
7. **无幻觉**：所有推理必须基于文档实际内容，不能编造不存在的线索或角色

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 输出中断/不完整 | 检查long output continuation的max轮次和token预算 |
| 续写断层 | 改进续写注入的上下文衔接提示词 |
| 压缩丢信息 | 调整predictive compaction的触发阈值，增加关键信息保留规则 |
| 超时 | 检查stream idle timeout是否足够（180s），是否需要更长 |

## T05: 学术论文技术演进深度分析

### 测试设计
**知识库**：bigtest（聚焦16篇PDF论文）
**提示词**：
> 请分析知识库中所有学术论文。要求：

## T08: 多模态内容分析（图片+音频+视频+SVG）

### 测试设计
**知识库**：bigtest
**提示词**：
> 请分析知识库中所有图片、音频、视频和SVG文件。要求：

> 3. 视频：描述视频内容，提取关键帧和对话
> 4. SVG：分析SVG中的文本内容、图形结构和可读信息（验证SVG走文本提取链路，非VLM图片处理）
> 5. 多模态关联：分析不同媒体文件之间是否存在关联（如同一事件的录音和照片）

7. **工具调用**：观察到 image_analysis、expand(L1含VLM描述) 的调用
8. **无幻觉**：图片描述与实际图片内容一致，音频转写与实际音频一致

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| VLM描述泛化 | 检查VLM模型配置和temperature，改进图片描述prompt |
| ASR空文本 | 检查Whisper服务状态，验证音频文件完整性 |
| 缺少关联分析 | 改进多模态关联分析提示，要求基于文件路径/时间戳关联 |
| 图片遗漏 | 先用wiki_browse按类型过滤图片文件，确认全覆盖 |
| SVG当作图片处理 | 检查 pipeline-strategies.ts 中 svg 映射为 "txt"（非 "image"），TextProcessor.HANDLED_TYPES 含 "svg" |
| SVG无文本提取 | 检查 TextProcessor 是否正确解析 SVG XML 结构并提取文本节点 |

## T10: 一致性验证——同一问题多次执行结果对比

### 测试设计
**知识库**：lbctest
**提示词**（连续执行3次，完全相同的提示词）：
> 请列出这个案件中所有涉及的资金转移记录，包括金额、日期、转出方、接收方。按时间顺序排列。

### 观察目标
1. **结果一致性**：3次执行的核心事实完全一致（相同的金额、日期、人名）
2. **结构一致性**：3次输出的表格/列表结构相似
3. **覆盖率一致性**：3次都找到相同数量的资金记录（允许排序差异）
4. **无矛盾**：3次输出之间不得存在互相矛盾的事实陈述
5. **差异分析**：如果存在差异（如措辞不同），差异仅限于非事实性内容

### 未达标时的优化方向
| 问题 | 优化方向 |
|------|---------|
| 事实不一致 | 加强anti-hallucination，要求先expand文档再总结 |
| 覆盖率波动 | 改进kb_search的topK和搜索策略，确保每次都能找到所有相关文档 |
| 结构差异大 | 使用structured_output工具固定输出格式 |
