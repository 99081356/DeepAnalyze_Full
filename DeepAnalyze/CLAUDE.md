# DeepAnalyze 项目规范

## 核心定位：通用 Agent 平台

**DeepAnalyze（DA）是一个通用 Agent 平台**，其通用 Agent 能力（对话、工具调用、多轮推理、任务编排等）不能低于任何主流 Agent 产品（如 Claude Code、OpenClaw 等）。DA 的核心 Agent 系统是一个完全通用的智能助手，能够处理任何类型的任务。

**知识库分析和报告生成只是 DA 的一个应用场景**，通过 Skill/Plugin 系统引导 Agent 在该场景下按要求工作。DA 不应在任何时候默认走"分析报告生成"路径 — 当用户没有知识库或没有上传文档时，DA 仍然是一个功能完整的通用 Agent。

具体要求：
- **默认行为是通用对话**：Agent 面对用户消息时，默认按通用助手模式响应，不预设"用户要分析文档"
- **知识库工具是可选能力**：kb_search、expand、doc_grep 等知识库专用工具在有知识库上下文时才被启用，不是 Agent 的默认工具集
- **报告生成不是默认输出**：Agent 默认以对话形式回答问题，仅在用户明确要求或 Skill 引导时才生成结构化报告
- **通用能力不能弱于专项 Agent**：代码编写、文件操作、命令执行、网络搜索等通用能力必须是一等公民

## 核心设计原则：通用性优先

DA 的核心 Agent 系统（提示词、工具、流程）必须对任何领域、任何类型的数据保持通用性。

**严格禁止**在以下位置包含领域特定内容：
- Agent 提示词（`agent-definitions.ts`）
- 工具描述和实现逻辑（`tool-setup.ts`）
- 路由处理和编排流程（`agents.ts`、`orchestrator.ts`）
- 前端组件和状态管理

**领域特定内容包括但不限于**：
- 特定领域的名称（如：剧本杀、论文、合同、法律等）
- 特定数据集的文件名或关键词（如：柯南之死、neurips 等）
- 特定测试数据的数值（如：222 个文档、16 篇论文等）
- 特定业务场景的流程分支

**正确做法**：
- 文档分类应基于**通用属性**（文件类型、目录结构、文件扩展名），而非内容语义
- 提示词中举例应使用**抽象泛化**描述（如"查看分类摘要和文档数量"），而非具体类别名
- 特定场景优化应通过 **Skill 系统**（插件化的自定义提示词+工具组合）实现，不在核心系统中

**判断标准**：修改前问自己——"如果用户没有知识库，或者知识库里全是医学文献/财务报表/工程图纸/法律卷宗，这个改动还有效吗？"如果答案是否定的，说明违反了通用性原则。

**常见错误**：在提示词中使用特定场景的举例（如"凶手判定"、"时间线推理"、"论文分析"）。这些应改为通用表述（如"责任归属"、"时序推理"、"学术文献分析"），或通过 Skill 系统实现。

## 工具描述原则：只描述能力，不做推荐

工具定义（`tool-setup.ts` 中的 description）和工具引导（`tool-guidance.ts`）必须严格遵循以下规则：

1. **只描述能力边界**：每个工具能做什么、不能做什么、覆盖率多少、局限性是什么
2. **不做使用推荐**：禁止"优先使用 X"、"X 适合其他工具无法覆盖的场景"、"应使用 X 而非 Y"等推荐性语言
3. **不规定固定流程**：禁止"A → B → C"的固定搜索顺序或决策树
4. **不偏袒任何工具**：专用工具（expand/doc_grep/kb_search）和通用工具（bash/grep/read_file）地位平等
5. **bash 是一等公民**：bash 拥有完整文件系统访问能力（grep批量搜索、python3编程处理、cat直接读取），与专用工具互补而非从属
6. **诚实标注局限性**：kb_search 的 topK 遗漏问题、doc_grep 只搜数据库不搜磁盘、expand 一次一个文档等
7. **不硬拦截合法操作**：不要在工具执行代码中拦截 Agent 的合法命令（如禁止 json.loads），只在真正失败时给提示

**常见违规**（发现即修正）：
- ❌ "请谨慎使用"、"适合其他工具无法覆盖的场景" → 这是推荐，不是能力描述
- ❌ "优先使用 expand 或 read_file，不要用 bash 解析文件" → 这是规定使用方式
- ❌ "搜索无结果时：doc_grep → grep → run_sql" → 这是固定顺序
- ❌ "读取文档内容优先使用 expand" → 这是偏袒
- ❌ 在 execute() 中硬拦截 json.loads → 限制合法操作
- ✅ "grep 可批量搜索全部文件；python3 可编程处理任意数据" → 能力描述
- ✅ "只搜索 wiki_pages 已编译内容，不含磁盘原始文件" → 局限性描述
- ✅ "topK=10 必然遗漏大量相关文档" → 诚实评估

## 优化必须遵循分层原则

参考 `docs/superpowers/specs/agent-optimization-design.md` 的 L1-L6 分层架构：
- **问题归属于最低的能解决的层级**
- L1（工具层）问题不要用 L2（提示词）解决
- L2 问题不要用 L5（编排）解决
- 领域特定优化一律归 L6（Skill/Plugin），不进入核心系统
- 任何优化修改必须先确认它对所有类型的知识库都有效，且对无知识库的通用场景也有效

## 优化通用性原则

**每次优化必须思考能否覆盖更多场景，让优化在更多场景受益。禁止针对某个用例做定制化优化。**

具体规则：
- **禁止用特定测试用例的上下文（如"推送所有写好的文件"）来描述通用机制** — 应抽象为通用表述（如"确认意图输出的内容都已完整输出"）
- **优化描述应覆盖正向和反向场景** — 例如：Agent 可能选择不输出低质量内容，这是合法的自主决策，系统不应强制所有输出必须推送
- **自检方法**：写完优化后问自己——"如果用户问的不是这个问题、没有知识库、Agent 做出不同的决策，这个优化还有意义吗？"
- **提示词和工具描述中的检查项应指导 Agent 自主判断，而非规定具体行为**

## Agent 输出完整性原则

Agent 的输出方式多样（文本流、push_content 卡片、write_file 文件），系统不强制每种输出都必须使用。Agent 根据任务需要自主决定输出方式和范围，但需遵守：
- **意图一致性**：如果 Agent 规划了多步输出（如分批推送多个文件），完成前应核对是否遗漏
- **自主取舍权**：Agent 可以选择不输出质量不达标的内容，但应在最终输出中说明省略了什么
- **不丢失**：系统层面确保已选择输出的内容不丢失、不截断；Agent 层面确保规划要输出的不遗漏

## 语言要求

- 所有设计文档、计划文档、需求文档必须使用**中文**编写
- 代码注释可以使用英文或中文，保持一致性即可
- Git 提交信息可以使用英文或中文

## 版本控制习惯

- **每次完成一批有意义的修改后，及时提交本地版本**，确保每个 commit 是一个可回退的原子单元
- 这样做的好处：如果后续修改引入问题，可以精确定位到哪个 commit 导致，方便 `git bisect` 或 `git revert`
- 不要积攒大量修改才提交，也不要每改一行就提交——以"一组逻辑相关的修改"为单位

### Commit 纪律

- **一个 commit 只做一件事**：清理归清理，优化归优化，修复归修复
- **清理 commit 禁止包含功能修改**：如果清理过程中发现需要调整代码逻辑（如新增改进、修复 bug、重构架构），必须放到单独的 commit
- **恢复操作必须验证**：任何"恢复"操作（如从旧版本恢复被误删的代码）完成后，必须 `git diff` 确认只恢复了目标内容，没有回退无关的功能改进。**禁止用文件级别的整体覆盖来恢复**（`git checkout prev -- path/`），应该用精确的编辑恢复被误删的部分
- **混合变更的分离**：如果一个大 commit 同时包含清理和优化，在提交前应该拆分为多个 commit，或者至少在 commit message 中明确列出所有变更类型，以便后续恢复时能精确操作

- 需求清单位于 `docs/superpowers/specs/2026-04-20-comprehensive-design/requirements-checklist.md`
- **每次完成涉及功能、架构、行为变更的开发任务后**，检查需求文档是否需要同步更新
- 更新原则：简洁准确，不膨胀。只修改受影响的条目，不添加冗余描述
- 以下情况**需要**刷新需求文档：
  - 需求理解变化（如设计调整、方案替代）

  - 纯 Bug 修复（不改变预期行为）
  - 代码重构（不改变外部行为）
  - 提示词细节调整（不影响需求定义）
  - 小的 UI 样式调整

## 基础设施

### 调测和测试环境
- **所有本地调测、测试、验证都在开发环境上进行**，使用 `python3 start.py` 启动
- **不要使用 Docker 容器进行调测**，Docker 镜像是用于客户部署的生产环境，和当前开发代码不同步
- 测试时使用 `--no-docker` 参数（跳过 Docker 容器管理，直连已运行的 PostgreSQL）
- 停止服务时，杀掉 `start.py` 启动的 node 进程和嵌入服务进程即可

### 启动方式
```
python3 start.py                        # 一键启动（Docker 容器 + 本地后端）
python3 start.py --dev                  # 开发模式（前端热重载）
python3 start.py --no-docker --skip-frontend --port 21000  # 调测常用命令
```

### 数据库：PostgreSQL + pgvector
- **唯一硬依赖**：PostgreSQL + `pgvector` 扩展
- **zhparser（中文分词）可选**：缺失时 FTS 降级为 default tokenizer，不影响启动
- 支持三种数据库来源（任选其一）：
  - Docker 预构建镜像 `pgvector/pgvector:pg17`（`docker-compose.dev.yml`）
  - 本地原生安装的 PostgreSQL（需自行安装 pgvector 扩展）
  - 云 PG（如 Neon / Supabase，需支持 pgvector）
- 连接信息：`.env` 文件中的 `PG_HOST`/`PG_PORT`/`PG_USER`/`PG_PASSWORD`

### Docker Compose 文件
- `docker-compose.dev.yml` — **开发用**（启动 PG + Ollama，后端本机跑 `npx tsx src/main.ts`）。PG 使用预构建 `pgvector/pgvector:pg17` 镜像，无需本地编译
- `docker-compose.yml` — **生产用**（前后端都跑容器）。PG 从 `config/pg-zhparser.Dockerfile` 构建（含 zhparser），如不需要 zhparser 可替换为 `image: pgvector/pgvector:pg17`

### 运行时
- Bun 运行 TypeScript，不需要编译到 `dist/`（`dist/` 目录不存在是正常的）
- 前端构建产物在 `frontend/dist/`

docker compose -f docker-compose.dev.yml down

## 数据层级使用原则（L0/L1/L2）

- **L0 (Abstract)**：文档摘要+标签+类型。极轻量，用于文档路由和快速分类
- **L1 (Structure)**：DocTags/Markdown 格式章节分块。Agent 分析工作的主要层级
- **L2 (Raw)**：Docling JSON 完整结构化原始数据。用于精确校验、锚点溯源、结构信息提取

关键认知：
- PDF/DOCX 的 L1 和 L2 **内容覆盖范围相同但格式不同**：L1 是可读 Markdown/DocTags，L2 是结构化 JSON
- 不应描述为"L2 与 L1 相同所以不需要展开"——两者格式不同，用途不同
- 日常分析阅读用 L1，精确校验和结构定位用 L2
- 图片 L1 含 VLM 预编译描述+OCR，优先用 expand 获取

## Skill 与 Plugin 设计原则

### Skill（轻量级场景经验）
- 单文件级别的经验指导，帮助 Agent 在特定场景下更高效地工作
- 可包含工具推荐、工作流建议、输出格式引导，但不应强制硬性约束
- 可被主 Agent 和子 Agent 通过 skill_invoke 按需调用
- 通用 Skill 不内置 report_generate——分析结果默认流式输出，仅在用户明确要求时生成报告
- **自适应深度引导**：引导模型根据任务复杂度自然决定输出深度，不硬性规定字数

### Plugin（重量级领域模块）
- 包含一组相关的 Skills + 自定义 Agents + 领域配置
- 如 judicial-analysis：证据链、时间线、实体网络、交叉验证、案件分析 + 验证器/提取器 Agent
- 领域特定需求通过 Plugin 实现，不污染通用系统

### 判断标准
- 如果一个能力是"在某个场景下如何更好地使用通用工具" → Skill
- 如果一个能力需要"自定义 Agent + 多个 Skills + 领域配置" → Plugin
- 如果修改只对一个领域有效 → Plugin；对所有领域都有效 → 通用 Skill 或核心系统

## Agent 输出引导原则

- **自适应深度**：引导模型根据问题复杂度和信息发现量自然决定输出深度
- 不硬性规定字数/篇幅（如"8000-20000字"），而是引导"充分回答，不多不少"
- 判断依据：发现的信息量、问题的复杂度、用户的期望
- 简单问题简明回答，复杂问题详尽展开
- 核心标准：充分解决问题、无遗漏关键信息、无幻觉、不注水

## 提示词修改原则：保留历史优化，有效融合

提示词中的很多内容是针对之前遇到的具体问题而优化的（如修正方式、幻觉判断、工具组合等）。在做新的设计和修改时：

- **不要用新设计直接覆盖旧内容**：旧提示词中看似不起眼的一句话，可能就是为了解决某个曾经反复出现的问题而加的
- **先分析旧内容的价值**：修改前先逐条分析旧提示词中每个要点存在的理由，识别出哪些是通用的、哪些是场景特定的
- **有效融合而非简单替换**：将旧内容中的通用价值点融合进新设计中，确保新方案的能力不弱于旧方案
- **通用化处理**：旧内容中场景特定的表述，抽象为通用表述后保留其核心逻辑（如"manifest.json"→"文件清单"，"凶手判定"→"责任归属"）

**自检方法**：修改完提示词后，将新旧版本逐条对比，确认旧版本中每个要点在新版本中都有对应，没有能力丢失。

## 调测和测试的深度验证标准（强制）

**所有测试验证必须深入检查实际内容质量，绝不能仅凭表面状态就判定通过。**

### 核心要求
- **禁止仅检查状态码和 status 字段**：文档 status=ready 不等于处理成功，必须验证实际内容
- **必须验证内容质量**：每个文件类型的处理结果都要检查实质内容是否正确、完整
- **必须检查边界情况**：空内容、错误信息被当作内容、VLM 失败但标记为就绪等

### 各类型文件的必查项

| 文件类型 | 必须验证的内容 | 禁止接受的"假成功" |
|---------|--------------|-------------------|
| **图片** (PNG/JPG) | VLM 视觉描述是否包含具体可读内容 | "[未配置VLM模型，跳过图像描述]"、"VLM不可用"、错误信息被当作描述 |
| **PDF** | Docling 解析是否提取到正文内容 | 只有页码/页眉、空文本 |
| **DOCX** | 文档结构+正文完整提取 | 仅提取到文件名或元数据 |
| **XLSX/CSV** | 表格元数据描述正确（行数+列数+样本行+文件路径） | 空表格、只有表头 |
| **音频** | ASR 转写文本有意义 | 空转写、乱码 |
| **视频** | 场景描述或对话转写有内容 | 空描述 |

### 测试流程标准
1. **上传验证**：文件存到磁盘正确位置，file_path 可访问
2. **处理验证**：处理完成后必须检查 wiki_pages 表中各 page_type 的 content 字段
3. **内容质量验证**：L0 摘要有意义（不是空或模板文本）、L1 结构化内容有实质信息
4. **前端验证**：浏览器中展开文档检查 L0/L1/L2 按钮是否可点击、内容是否显示
5. **回归验证**：每次修改后重新运行所有 e2e 测试确认无回归

### VLM 相关验证
- VLM 失败的图片**不应**标记为 ready，应在文档中标注 VLM 异常
- 切换 VLM 提供商后必须抽样检查输出质量（至少检查 5 张图片的实际描述内容）
- VLM 配置变更（如从 Qwen 切到 MiniMax）必须保持向后兼容，原有配置界面不受影响

## 测试文件组织规范

- **所有测试文件统一放在 `tests/` 目录下**，包括：
  - 单元/集成测试：`tests/*.test.ts`、`tests/**/*.test.ts`
  - E2E 测试：`tests/e2e/*.spec.ts`
  - 测试辅助脚本：`tests/*.mjs`、`tests/helpers/`
  - 基准测试：`tests/benchmark/`
  - 测试夹具：`tests/e2e/fixtures.ts`
- **禁止在 `src/` 目录下创建测试文件**（`src/**/__tests__/` 下的旧测试逐步迁移到 `tests/`）
- **禁止在项目根目录散放测试脚本**
- Playwright 配置文件 `playwright.config.ts` 放在项目根目录

## 弹性执行原则（禁止硬性执行约束）

Agent 执行过程中不应设置硬性的轮次、时间、长度等限制。所有约束都应是弹性的、根据任务动态调整的：

- **轮次控制**：使用 estimateTaskComplexity 动态估算 + ProgressTracker 弹性管理（有进展自动延长，无进展引导换策略）
- **输出长度**：使用 max_output_tokens 分级恢复机制（截断后自动续写），不硬性限制单次输出
- **执行时间**：根据任务复杂度动态分配，而非统一超时
- **Skill 执行**：Skill 定义不包含 maxTurns 硬性约束，Skill 是经验指导，不是执行限制

禁止事项：
- Skill 定义中不设 maxTurns 硬性值
- 不硬性规定输出字数范围
- 不硬性规定工具调用次数上限
- 不硬性规定执行时间上限

### 测试执行要求
1. **逐个执行**：一个测试用例一个测试用例地跑，不并行，不批量
2. **全面观察**：每个测试必须同时观察前端和后端的完整表现
3. **观察内容包括**：
   - 后端日志（`/tmp/da_debug*.log`）中的所有输出
   - 前端浏览器控制台错误（Playwright consoleErrors）
   - 前端页面截图（Playwright takeScreenshot）
   - API 响应状态和内容
   - Agent 的工具调用记录和输出质量
4. **发现即修复**：每个测试发现的问题必须立即系统分析根因并修复
5. **修复验证**：每个修复后重新运行受影响的测试确认通过
6. **问题记录**：每个测试的结果、发现的问题、修复方案都要记录

### 测试深度观察标准（强制）

**测试不仅是验证"跑通"，更要深入检查以下5个维度，发现潜在问题必须修复：**

1. **执行过程合理性**：
   - 工具调用顺序是否合理（先搜索后展开，先浏览后分析）
   - Agent是否做了不必要的重复操作（多次搜索同一关键词、重复展开同一文档）
   - 工具参数是否正确（kbId是否正确、搜索关键词是否恰当）
   - Agent是否高效利用了已有信息（不重复搜索已找到的内容）

2. **执行过程高效性**：
   - 工具调用次数是否合理（不过多也不过少）
   - 是否有不必要的"兜底"操作（搜索无果后盲目尝试其他工具）
   - 上下文使用效率（是否有大量重复内容占用context）
   - 执行时间是否合理（不因效率问题导致超长运行）

3. **执行过程健壮性**：
   - 后端日志是否有 WARNING 或 ERROR 级别日志（即使是可恢复的）
   - 是否有未捕获的异常（`Uncaught exception`、`unhandledRejection`）
   - 前端控制台是否有非预期的错误（排除已知的 favicon/ResizeObserver）
   - API 响应是否有非200状态码
   - 数据库操作是否有异常（JSON解析失败、写入超时）
   - 流式输出是否有中断或异常（SSE断连、事件丢失）

4. **结果完整性**：
   - 是否所有要求的子任务都有输出（无遗漏）
   - push_content 卡片内容是否完整（无截断、无空白）
   - 工具调用结果是否都被合理利用（不丢弃有价值的搜索结果）
   - 最终输出是否覆盖了关键内容维度

5. **无幻觉**：
   - 关键事实（人名、金额、日期、数据）是否与原文一致
   - 引用的文档是否存在（不引用不存在的文档）
   - 技术术语、指标数据是否准确
   - 推理过程是否有合理的证据支持

**执行原则**：
- 发现小问题也要修，不能因为"功能正常"就忽略潜在隐患
- 后端日志中的 WARNING 即使不影响功能，也要分析是否需要优化
- 前端控制台的错误即使不致命，也要确认不影响用户体验
- 工具调用的低效模式（如重复搜索）如果频繁出现，需要优化提示词

- **P0（必做）**：T01, T02, T03, T11, T24, T29, T50
- **P1（重要）**：T06, T10, T15, T17, T19, T22, T38, T41, T45, T48
- **P2（补充）**：其余测试项

### 单个测试执行流程
1. 阅读测试设计文档，理解观察目标
2. 检查后端服务是否正常运行
3. 执行测试（通过 Playwright 或手动 API 调用）
4. 分析后端日志：检查错误、警告、工具调用记录
5. 分析前端截图：检查渲染正确性、控制台错误
6. 对照观察目标逐项验证
7. **深度分析**：检查执行过程合理性、高效性、健壮性（按上述5维度）
8. 发现问题 → 分析根因 → 按优化分层原则修复 → 重新验证
9. 记录测试结果和修复内容
